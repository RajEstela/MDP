Tuning (based on first real test - dark image, false positives on
legs/chairs, and the printout format)
----------------------------------------------
1. FALSE POSITIVES (legs/chair flagged as targets): the confidence
   threshold defaulted to 0.12 (12%) originally, way too permissive.
   Default is now 0.2 in pc_infer_server.py. Adjust live without editing
   code:
       python3 pc_infer_server.py --serve --port 5006 --conf 0.3
   Raise further (e.g. 0.4, 0.5) if false positives continue; lower it if
   real targets stop being picked up. Applies to both --serve and
   standalone modes.

2. DARK CAMERA IMAGE (Pi Camera Module / CSI): added best-effort
   --brightness and --exposure flags to pi_stream_frames.py.
   CONFIRMED WORKING VALUE: --brightness 60 - tested and gives a
   well-exposed image (no longer dark/washed out), now the default used
   in the Quick Start command above:
       python3 pi_stream_frames.py --host 192.168.3.13 --port 5006 --brightness 60
   The script prints back what value the driver actually reports after
   setting it - if it doesn't change, this OpenCV/V4L2 property isn't
   well supported by the bcm2835-v4l2 driver on this OS, and the more
   reliable fallback is to set it directly via v4l2-ctl instead:
       v4l2-ctl --list-ctrls -d /dev/video0     # see available controls + ranges
       v4l2-ctl -d /dev/video0 --set-ctrl=brightness=60
   (exact control names/ranges vary - list-ctrls shows what's actually
   available on this specific camera/driver.)
   Sharpness: the confirmed snapshot at --brightness 60 looked sharp
   (legible "8" digit, clear wood-grain background texture) - no evidence
   of blur, so no sharpness-specific tuning needed for now. Focus was
   separately confirmed sharp at 20cm too, so this isn't a lens issue.

3. FLICKERING BOUNDING BOX at 50cm (box appears/disappears frame to
   frame): this is normal single-frame confidence variance - a real
   target sitting right at the edge of the model's confidence threshold
   will flicker in and out as lighting/angle/frame noise shifts slightly
   frame to frame. Two options once you're ready to address it:
     - Quick lever: lower --conf slightly (e.g. 0.15) if the target itself
       is correct but marginal - trades off against false positives from
       item #1.
     - Better fix (not yet implemented, revisit once lighting/focus fully
       settled): add debounce logic that only reports a detection once
       the SAME target ID appears consistently across several consecutive
       frames, instead of reacting to any single frame's best guess. This
       smooths out flicker AND helps filter one-off background
       misclassifications (e.g. the cupboard false positive at 20cm).

4. WIRE PROTOCOL CLARIFICATION (what "IMGRES,0,12" means):
   Format: IMGRES,<obstacleId>,<targetID>
     - obstacleId: which obstacle number the robot is currently in front
       of. Currently always "0" because this is a camera-only test with
       no real obstacle sequence yet - it becomes meaningful once wired
       to P1/P4's real algorithm/hub code (each obstacle gets its own ID
       during the actual run). This is still sent on the wire for that
       future integration, just no longer shown in the friendly console
       print since it's not meaningful yet.
     - targetID: the recognised image ID from the confirmed table (e.g.
       12 = Alphabet W, 100 = Bullseye).
   The confidence number (e.g. 0.12 in "conf 0.12") IS the model's
   confidence level - 0.12 = 12% confident. Combined with the old 0.12
   threshold, this meant the lowest possible passing detections were
   being shown, hence the false positives - fixed by #1 above.

5. PRINTOUT FORMAT (updated): both scripts now show the image's
   description alongside its ID, and confidence as a percentage, e.g.:
       Detected: ID 12 (Alphabet W)  [58% confidence]
       Detected: ID 100 (Bullseye)  [71% confidence]
   The raw obstacle_id placeholder is no longer shown in this line (see
   #4), but is still sent correctly in the underlying IMGRES wire message
   for the real robot logic to use later.

----------------------------------------------

whenever it gets updated (new flags, bug fixes, etc.), the copy already
sitting on the RPi does NOT update itself. Re-copy it before the next test
run whenever it's changed:

(adjust the destination path to wherever you're actually running it from
on the RPi - confirmed as ~/share/ based on the last test session.)

Troubleshooting (if the Quick Start run above doesn't work)
----------------------------------------------
- Connection refused: wrong IP, or Mac/RPi not on same subnet (check both
  are on MDPGrp3), or macOS firewall blocking - the first time
  pc_infer_server.py --serve binds a listening socket, macOS may prompt
  "Do you want the incoming network connection to be accepted?" - click
  Allow. Try "ping 192.168.3.13" from the RPi to sanity-check the route.
- Very laggy/choppy video: reduce --jpeg-quality (default 70) or
  --resize-width (default 416) on the pi_stream_frames.py side - smaller
  frames = faster over Wi-Fi.
- No detections ever, even up close: confirm lighting/distance matches the
  20-50 cm standoff the model was trained/tested at.
- If the RPi camera doesn't open at all: it likely needs enabling first:
      sudo raspi-config
      # Interfacing Options -> Camera -> Enable
      sudo reboot
- No-display camera check (no monitor, no X11 forwarding on the RPi):
      python3 pi_stream_frames.py --snapshot snapshot.jpg
  then from the Mac: scp pi@192.168.3.3:~/snapshot.jpg . and open it.

What --serve and --host actually mean
----------------------------------------------
--serve (on pc_infer_server.py, runs on the Mac): by default (no flags),
pc_infer_server.py just opens the Mac's OWN webcam and runs YOLO on it
(same as detect_live_v2_test.py). --serve switches it into network-
listener mode instead - it opens a TCP socket and waits for something (the
RPi) to connect and send frames over the network.

--host (on pi_stream_frames.py, runs on the RPi): tells the RPi where to
send its camera frames to - the IP address of whichever machine is running
pc_infer_server.py --serve. Without --host, the script just does
--local-preview (or --snapshot) and never touches the network.

Put together:
  Mac (--serve) = the server: sits waiting, does the actual YOLO
  inference, shows the annotated window.
  RPi (--host <mac-ip>) = the client: actively connects out to the Mac
  and pushes its camera frames there.

This is also why the order matters: the server (--serve) needs to be up
and listening BEFORE the client (--host) tries to connect, otherwise the
RPi gets "connection refused."
